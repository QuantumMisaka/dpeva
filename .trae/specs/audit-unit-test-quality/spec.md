# 单元测试深度审查 Spec

## Why
当前仓库已经形成较完整的单元测试体系，但覆盖率门禁、测试命名一致性、Mock 隔离边界、性能基线与文档同步性仍存在不确定性。需要基于 `docs/` 与 `src/` 的真实预期，对 `tests/` 执行一次可追溯、可落地的深度审查，形成后续整改依据。

## What Changes
- 对 `docs/` 中与设计、接口、业务需求、开发测试约束相关的文档进行梳理，提炼模块预期行为、边界条件与测试责任
- 对 `src/` 的核心模块逐文件建立“公开 API / 核心算法 / 异常路径 / 性能要求”审查基线
- 对 `tests/` 中的单元测试逐条审查覆盖率、正反向与边界用例、Mock 隔离、断言精度、命名规范、fixture 与测试数据质量
- 收集覆盖率、执行时长、Lint 与类型检查等证据，识别缺失、错误或过时测试并定位到具体文件与行号
- 输出一份详细审查报告，包含问题分级、优先级、修复建议、回归验证步骤与持续集成稳定性建议

## Impact
- Affected specs: 单元测试质量审计、测试治理、覆盖率门禁、CI 稳定性
- Affected code: `docs/`、`src/`、`tests/`、`.github/workflows/`、`pyproject.toml`、`pytest.ini`

## ADDED Requirements
### Requirement: 预期行为基线梳理
系统 SHALL 先基于项目文档与核心源码建立单元测试审查基线，再对测试用例进行逐项比对。

#### Scenario: 从文档与源码提炼测试基线
- **WHEN** 审查开始执行
- **THEN** 必须先阅读 `docs/` 中与设计、接口、需求、开发与测试规范相关的文档
- **THEN** 必须逐文件分析 `src/` 的关键模块，提取公开 API、核心算法、输入输出、异常路径与性能约束
- **THEN** 审查报告必须显式区分“文档预期”“源码真实行为”“测试现状”三类信息

### Requirement: 单元测试覆盖与质量核查
系统 SHALL 基于基线对 `tests/` 中的单元测试执行覆盖率与质量核查，并输出可定位的问题清单。

#### Scenario: 覆盖率与用例质量可核验
- **WHEN** 审查 `tests/` 中的单元测试
- **THEN** 必须检查语句、分支、函数、行四级覆盖率是否达到 90% 目标，若未达标则标注差距与影响模块
- **THEN** 必须检查每个公开 API 与核心算法是否具备正向、负向与边界测试
- **THEN** 必须检查 Mock 是否有效隔离数据库、网络、文件系统、外部命令或作业调度等依赖
- **THEN** 必须检查断言是否精确，是否存在过度宽松、重复断言或无效断言
- **THEN** 必须检查测试命名是否符合“被测单元_条件_预期结果”模式，或指出偏离与整改方式

### Requirement: 测试数据与执行效率审查
系统 SHALL 对测试数据、fixtures 与执行效率进行独立评估，并给出稳定性建议。

#### Scenario: 测试数据与性能风险被识别
- **WHEN** 审查 fixtures、样例数据与执行表现
- **THEN** 必须检查测试数据是否符合真实业务分布，是否存在硬编码路径、魔法值或脆弱样例
- **THEN** 必须评估单测平均耗时是否低于 50ms、总套件是否在 30 秒内完成，并识别长尾测试与潜在资源泄漏
- **THEN** 若受环境限制无法完整证明某项指标，报告必须说明限制条件、当前证据与补充验证步骤

### Requirement: 测试代码规范一致性审查
系统 SHALL 将测试代码视为生产资产，审查其规范一致性与可维护性。

#### Scenario: 测试代码遵循项目规范
- **WHEN** 审查测试实现
- **THEN** 必须检查测试代码是否通过既有 Lint 规则，并评估是否具备类型检查与复杂度约束
- **THEN** 必须标识违反项目编码规范、结构过度复杂或可读性不足的测试文件

### Requirement: 审查报告与整改建议交付
系统 SHALL 输出可直接用于后续整改与回归验证的正式审查报告。

#### Scenario: 报告可直接驱动后续整改
- **WHEN** 审查结论形成
- **THEN** 报告必须包含审查结论、覆盖率证据、问题清单、优先级排序、修复建议与回归验证步骤
- **THEN** 每个问题必须尽量定位到具体文件与行号，并说明影响范围与建议动作
- **THEN** 报告必须说明关键功能如何在持续集成流水线中稳定通过

## MODIFIED Requirements
### Requirement: 审查证据完整性
系统 SHALL 在输出结论前收集代码阅读、覆盖率统计、测试运行、静态检查与配置核查证据，避免仅基于抽样印象形成判断。

#### Scenario: 所有关键结论均有证据支撑
- **WHEN** 输出最终审查报告
- **THEN** 覆盖率、性能、规范性与 CI 结论必须引用对应的测试、配置或运行证据
- **THEN** 对无法自动验证的部分必须标记为限制项，不得伪造结果
