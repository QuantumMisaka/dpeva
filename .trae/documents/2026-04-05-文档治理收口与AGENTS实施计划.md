# 文档治理收口与 AGENTS 实施计划

## Summary

本计划用于收口上一轮“文档治理审阅与 AGENTS 目录化改造”工作的剩余工程项，并把结果推进到可直接实施 `AGENTS.md` 改造的状态。

本轮执行目标分为三部分：

1. 依据当前 Sphinx 文档治理模式，确认并治理 `docs/reference` 与 `docs/source/reference` 的双维护问题；
2. 确认并补齐 `docs/reports/2026-04-05-Code-Review-Agents-Docs-Governance.md` 中仍未完全闭环的文档治理事项，使之真实符合 spec/checklist/tasks 的完成口径；
3. 确认 `docs/plans/2026-04-05-AGENTS-Docs-Governance-Plan.md` 已具备实施前提，并在必要时补充到“可直接改写 `AGENTS.md`”的粒度。

## Current State Analysis

### 1. `docs/reference` 与 `docs/source/reference` 现状

已通过只读核查确认：

- `docs/source/reference` 是指向 `../reference` 的符号链接；
- `docs/source/index.rst` 的主导航实际指向 `docs/source/reference/index`；
- 因此从仓库结构上看，`reference` 不是“两套正文双维护”，而是**`docs/reference` 单源 + `docs/source/reference` Sphinx 路径别名**。

结论：

- 这里需要治理的不是“拆双维护”，而是**确认并记录单源事实**；
- 同时要避免后续文档或报告继续把该结构误写成“`docs/reference` 与 `docs/source/reference` 两套正文”。

### 2. `docs/build` 现状

已通过只读核查确认：

- 仓库根 `.gitignore` 已包含 `build/`；
- 当前工作区实际存在大量 `docs/build/**` 产物；
- 仅从文件系统无法确认这些产物是否仍被 Git 跟踪。

结论：

- “是否需要清理 `docs/build`”要拆成两个问题：
  - 忽略规则是否存在：**已存在**；
  - 历史跟踪是否已解除：**仍需 Git 级核查**。
- 执行阶段应先做 `git ls-files docs/build` 只读确认，再决定是否删除被跟踪的构建产物。

### 3. 审阅报告与 spec 完成度现状

已通过只读核查确认：

- `.trae/specs/audit-agents-doc-governance/spec.md`、`tasks.md`、`checklist.md` 均已勾选完成；
- `docs/reports/2026-04-05-Code-Review-Agents-Docs-Governance.md` 已记录已修复事项与后续治理项；
- 但报告中仍保留两类尚未彻底闭环的内容：
  - `docs/reference` / `docs/source/reference` 双维护风险；
  - `docs/build` 是否只是本地缓存、还是仍存在版本库残留，尚未被证据化结论关闭。

同时还发现一类“已从初步修复转为第二阶段治理”的项目：

- `docs/guides/developer-guide.md`、`docs/governance/traceability/feature-doc-matrix.md` 等文档仍将 `docs/source/api/config.rst` 作为直接文本路径暴露给读者；
- 这不一定违反当前 spec，但会削弱“面向读者的稳定入口”与“仓库内部实现路径”之间的边界。

结论：

- 当前 spec 文档状态是“按第一轮定义已完成”；
- 但若要满足你现在提出的“确认治理已全部完成”这一更高标准，需要继续补做一轮治理收口，并更新审阅报告中的完成状态与剩余风险说明。

### 4. `AGENTS` 计划可实施性现状

已通过只读核查确认：

- `docs/plans/2026-04-05-AGENTS-Docs-Governance-Plan.md` 已提供：
  - 目标；
  - 改造前提；
  - 设计原则；
  - 建议结构；
  - 保留/压缩/删除/新增项；
  - 与文档系统的联动矩阵；
  - 精简骨架；
  - 实施顺序与验收标准。
- 当前 `AGENTS.md` 仍是旧版“操作须知型”文档，未开始实际改写。

结论：

- 该计划在结构上**已经具备实施条件**；
- 但在真正改写 `AGENTS.md` 之前，应先完成本计划前两部分的治理收口，避免继续把未收敛的文档入口重新索引进 `AGENTS.md`。

## Proposed Changes

### A. 确认 `reference` 单源事实并清理误判

#### 目标

确认 `docs/reference/` 是项目声明的查表类权威来源，而 `docs/source/reference/` 只是指向它的 Sphinx 路径别名；清理文档中把它误写成“双维护”的表述。

#### 文件范围

- `docs/reference/README.md`
- `docs/reference/validation.md`
- `docs/reference/upstream-software.md`
- `docs/source/reference/index.rst`

#### 决策

- `docs/reference/*.md` 作为正文单一权威来源；
- `docs/source/reference` 作为指向 `../reference` 的既有符号链接保留；
- 不在本轮扩展到 `guides/`、`policy/`、`architecture/` 全树去重，只解决本次用户明确点名的 `reference` 误判问题。

#### 实施方式

实施方式以“确认结构 + 更正文档结论”为主：

1. 核查并记录 `docs/source/reference -> ../reference` 的符号链接事实；
2. 保持 `docs/reference/*.md` 为正文单源，不额外引入包装层复杂度；
3. `docs/source/reference/index.rst` 继续作为 toctree 路径别名使用；
4. 在报告与计划文档中更正此前关于“reference 双维护”的误判。

#### 为什么这样做

- 能与当前仓库真实结构保持一致；
- 不引入不必要的 include/包装层复杂度；
- 不破坏 `docs/source/index.rst -> reference/index` 的既有导航；
- 保留 `docs/reference/` 作为对开发者声明的权威目录。

### B. 关闭 `docs/build` 策略不确定性

#### 目标

把“`docs/build` 到底要不要清理”的问题从猜测变成证据化结论。

#### 文件范围

- `.gitignore`
- `docs/reports/2026-04-05-Code-Review-Agents-Docs-Governance.md`
- 视结果而定：`docs/build/**`（仅当确认仍被 Git 跟踪时）

#### 决策

- 执行阶段先运行只读检查：`git ls-files docs/build`
- 分两种路径处理：
  - 若无输出：说明 `docs/build` 只是本地忽略产物，不改仓库文件，只在报告中明确“忽略规则已存在，当前无需治理”；
  - 若有输出：说明存在历史遗留跟踪，需要删除被跟踪产物，并在报告中记录“已解除版本控制污染”。

#### 为什么这样做

- 当前 `.gitignore` 已经包含 `build/`，没有必要在未确认问题前再盲目改 ignore 规则；
- 真正需要闭环的是“历史跟踪状态”，而不是“再加一条 ignore”。

### C. 让审阅报告与 spec 完成状态重新对齐

#### 目标

把报告中的“已完成治理”与仓库当前事实重新对齐，让 `spec/checklist/tasks` 的完成状态不只是纸面成立。

#### 文件范围

- `docs/reports/2026-04-05-Code-Review-Agents-Docs-Governance.md`
- 如有必要：`docs/reports/README.md`
- 如有必要：`.trae/specs/audit-agents-doc-governance/checklist.md`
- 如有必要：`.trae/specs/audit-agents-doc-governance/tasks.md`

#### 决策

- 不轻易回滚现有“已完成”状态；
- 优先通过补完治理动作，使报告与 checklist 重新成立；
- 仅当出现新的阻断性事实时，才回写 spec/checklist/tasks 的状态描述。

#### 具体收口项

1. 在报告中新增“第二阶段治理闭环”或“Closure Status”小节；
2. 明确写出：
  - `reference` 双维护是否已被消除；
  - `docs/build` 是否已确认无需仓库治理，或已清理历史跟踪；
  - 是否仍存在仅作为后续优化、但不影响本轮完成判定的事项；
3. 将“已完成”和“后续建议”边界重写得更精确，避免把未落地项继续写成完成项。

### D. 统一面向读者的稳定文档入口

#### 目标

在 AGENTS 改造前，先把最关键的配置/Reference 跳转统一成稳定入口，避免继续把 `docs/source/api/config.rst` 这类实现层路径直接暴露给普通读者。

#### 文件范围

- `docs/guides/developer-guide.md`
- `docs/governance/traceability/feature-doc-matrix.md`
- `docs/reference/README.md`
- 必要时联动的 `docs/guides/*.md`

#### 决策

- 对开发者可见的“权威配置入口”统一表达为：
  - API Reference（渲染后的稳定文档入口）
  - `docs/reference/validation.md`
- 仅在治理/维护文档中才允许保留 `docs/source/api/config.rst` 这样的构建层路径提示；
- 若某页承担的是“面向构建维护者”的职责，可继续保留源路径，但要显式标注“这是 Sphinx 源文件位置，不是读者入口”。

#### 为什么这样做

- 这一步是 `AGENTS.md` 可实施的真正前置条件之一；
- `AGENTS` 需要指向稳定入口，而不是把 source-root 细节带到第一层导航里。

### E. 确认 AGENTS 计划达到“可直接执行”粒度

#### 目标

在不立即改写 `AGENTS.md` 的前提下，把计划文件补强到执行者无需再补充决策。

#### 文件范围

- `docs/plans/2026-04-05-AGENTS-Docs-Governance-Plan.md`
- `AGENTS.md`
- 如有必要：`docs/README.md`
- 如有必要：`docs/guides/docs-governance-quickstart.md`

#### 决策

- 只要前四部分治理完成，就直接执行 `AGENTS.md` 重写；
- `AGENTS` 改写时严格遵循计划文件已有六模块结构；
- 若计划文件缺少“每一节具体跳转到哪个现有文档”的最终落点，就在实施前先补上精确路径映射。

#### 实施步骤

1. 对照当前 `AGENTS.md` 与计划文件的“保留/压缩/删除/新增项”逐段映射；
2. 给每个最终模块绑定一个现有权威文档入口；
3. 若 `docs/README.md` 或 `docs-gov quickstart` 与新 AGENTS 首屏明显重复，则同步瘦身其中一方，而不是把重复保留到两边。

## Assumptions & Decisions

### 已确认事实

- `docs/source/reference` 是到 `docs/reference` 的符号链接，不是独立正文目录；
- `.gitignore` 已覆盖 `build/`；
- `docs/build/**` 在工作区实际存在；
- report/spec/checklist/tasks 已处于“第一轮完成”状态；
- `AGENTS` 计划结构上已可实施，但应在文档治理收口后再执行。

### 本计划的边界决策

- 本轮重点不是全树重构 `docs/` 与 `docs/source/` 的关系；
- 本轮只对用户点名的 `reference` 双维护做结构性治理；
- `docs/build` 是否删除，取决于 Git 跟踪状态，而不是仅取决于目录是否存在；
- 若执行中发现 `docs/source/reference` 包装方案与 MyST/Sphinx 不兼容，则优先选择“共享正文片段”方案，而不是回退为继续双写。

### 不纳入本轮的内容

- `guides/`、`policy/`、`architecture/` 全量去重；
- 大规模归档历史文档；
- 直接改写 `AGENTS.md` 本体之前的任何额外产品方向变更。

1. 结构与策略验证

2. 内容验证

   - `docs/reports/2026-04-05-Code-Review-Agents-Docs-Governance.md` 明确记录治理闭环状态
   - `docs/plans/2026-04-05-AGENTS-Docs-Governance-Plan.md` 对 `AGENTS.md` 改写给出可直接执行的落点
   - 若进入 AGENTS 实施阶段，改写后的 `AGENTS.md` 应能通过文档构建和人工抽检
