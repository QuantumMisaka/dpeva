## Assumptions & Decisions

- 决策 1：整改顺序按 P1 → P2 → P3 执行，本轮优先落地审查报告中的 F1-F8，不扩展到新的功能开发。
- 决策 2：CI 先建立“最小但强约束”的 Python 质量门禁，再考虑更重的类型检查与覆盖率阈值；`mypy` 在第一阶段作为可选告警项设计，不默认阻断，除非仓库现状足够稳定。
- 决策 3：文档治理以“活跃文档必须受治理”为原则；`archive` 继续排除，但 `plans` 和 `reports` 从豁免名单中移除。
- 决策 4：根 README 只保留稳定可读入口，不再把 `docs/source/*.rst` 作为面对最终读者的链接目标。
- 决策 5：发布辅助链路采用“脚本 + 文档 + 技能”三者同源更新，不允许只修其中一个。
- 决策 6：测试整改优先提升便携性与契约一致性，不在本计划中扩大到大规模重写测试架构。

## Proposed Changes

### Phase A：建立最小 Python 质量门禁

#### A1. 新增统一源码质量 workflow

- 文件：
  - 新增 `.github/workflows/python-quality.yml`
  - 视需要更新 `.github/CODEOWNERS`
- What：
  - 在 `push` / `pull_request` 的 `main` 与 `develop` 上，对 `src/**`、`tests/**`、`pyproject.toml`、`AGENTS.md`、`scripts/**`、相关 workflow 改动触发统一质量检查。
  - 首批 job 至少包含：
    - `ruff check .`
    - `pytest tests/unit`
  - 预留 `mypy` job 结构，但默认不阻断，或仅在显式启用时运行。
- Why：
  - 把 [AGENTS.md](file:///home/pku-jianghong/liuzhaoqing/work/ft2dp-dpeva/dpeva/AGENTS.md#L22-L34) 与 [pyproject.toml](file:///home/pku-jianghong/liuzhaoqing/work/ft2dp-dpeva/dpeva/pyproject.toml#L40-L47) 中承诺的质量标准真正落到 CI。
- How：
  - 使用 `actions/setup-python` 的 pip cache。
  - 统一安装 `pip install -e .[dev]`。
  - 为 job 增加 `timeout-minutes` 与最小 `permissions: contents: read`。
  - 若 `ruff check .` 受 `.trae/skills` 现有问题影响，则在执行前明确决定：
    - 方案 A：修复 `.trae/skills` 下的现有 Ruff 问题并保留全仓门禁。
    - 方案 B：将 lint 范围收敛为项目受控路径，并同步更新 [AGENTS.md](file:///home/pku-jianghong/liuzhaoqing/work/ft2dp-dpeva/dpeva/AGENTS.md#L28-L30) 使规则一致。
  - 默认采用方案 B 作为第一阶段落地，避免 CI 初始即被工作区资产阻断。

#### A2. 收敛文档 workflow 职责

- 文件：
  - `.github/workflows/docs-check.yml`
  - `.github/workflows/doc-lint.yml`
  - `.github/workflows/docs.yml`
  - `.github/workflows/docs-deploy.yml`
- What：
  - 区分“文档检查”和“文档部署”两类职责。
  - 清理重复构建与重复安装。
- Why：
  - 降低维护漂移与 CI 成本。
- How：
  - 保留 `docs-check.yml` 负责 PR / push 上的严格构建、关键产物检查、可选 linkcheck。
  - 保留 `doc-lint.yml` 负责治理脚本检查。
  - 合并 `docs.yml` 与 `docs-deploy.yml` 的重复构建逻辑，保留唯一发布链路。
  - 为 docs 相关 workflow 增加 cache、concurrency、timeout、明确 permissions。

### Phase B：修复文档治理与对外入口断链

#### B1. 收紧治理脚本

- 文件：
  - `scripts/doc_check.py`
  - `scripts/check_docs_freshness.py`
  - 视需要更新 `docs/policy/quality.md`
- What：
  - 让 `docs/plans/`、`docs/reports/` 进入 active 文档治理范围。
  - 明确 owner 覆盖是否为真正阻断项。
- Why：
  - 消除“规则要求严格，但脚本默认放过”的矛盾。
- How：
  - 从 `IGNORE_DIRS` 中移除 `plans`、`reports`。
  - 把 owner 检查从 non-blocking 提升为可配置阻断，默认在 CI 中阻断 active 文档缺 owner 的情况。
  - 在 freshness 检查中保留 `archive` 与 `_templates` 排除，但将 `plans`、`reports` 纳入检查，必要时允许用命令行参数控制例外。
  - 清理 `doc_check.py` 中重复的 `if "archive" in path.parts` 判断。

#### B2. 重写根 README 的 Documentation 区块

- 文件：
  - `README.md`
  - 如有必要，`docs/README.md`
- What：
  - 把根 README 改为面向最终读者的稳定导航入口。
- Why：
  - 当前入口已经把占位链接、源文件路径和漂移路径暴露给读者。
- How：
  - 删除 `https://dpeva.readthedocs.io` 占位标记文案。
  - 不再链接 `docs/source/api/config.rst`。
  - 改为指向稳定的 Markdown 入口，例如：
    - `docs/README.md`
    - `docs/guides/quickstart.md`
    - `docs/guides/installation.md`
    - `docs/guides/cli.md`
    - `docs/reference/upstream-software.md`
    - `docs/reports/README.md`
  - 清理已漂移的计划/报告链接，优先指向索引页而非单个历史文件。

### Phase C：收敛发布辅助知识与治理脚本入口

#### C1. 修复 release helper 链路

- 文件：
  - `scripts/release_helper.py`
  - `.trae/skills/release-helper/SKILL.md`
  - `docs/guides/developer-guide.md`
- What：
  - 对齐脚本路径、技能说明和开发文档中的版本维护规则。
- Why：
  - 当前三者事实不一致，会误导维护者执行错误路径和错误纪元。
- How：
  - 将 `release_helper.py` 中的 `DEV_GUIDE` 修正为 `docs/guides/developer-guide.md`。
  - 将 skill 中的 `Current Era (v0.4.x)` 更新为当前真实纪元。
  - 明确脚本的职责边界：只做版本号更新，还是同时维护开发者指南；若保持“只更新版本号”，则 skill 中不得宣称已自动同步文档。
  - 在 `developer-guide.md` 中补一段 release helper 的权威使用说明，作为单一维护入口。

#### C2. 决定脚本归宿并接入自动化

- 文件：
  - `scripts/gate.sh`
  - `scripts/verify_docs.sh`
  - `scripts/audit.py`
  - 相关 workflow
- What：
  - 确定哪些脚本进入 CI，哪些仅保留本地维护用途。
- Why：
  - 减少“存在但无人执行”的脚本噪声。
- How：
  - `audit.py`：纳入 Python 质量或单独的代码审计 job。
  - `gate.sh`：作为本地便捷入口保留，但其命令集合需与 workflow 保持一致。
  - `verify_docs.sh`：若职责与 docs-check workflow 完全重合，则要么简化为本地辅助脚本，要么下线冗余逻辑。

### Phase D：提升测试便携性与源码可诊断性

#### D1. 移除测试中的本机路径依赖

- 文件：
  - `tests/integration/test_labeling_rotation_bug.py`
  - 相关测试数据 README 或 fixture 辅助文件
  - `tests/unit/UNIT_TESTS.md`
- What：
  - 改用仓库相对路径、环境变量或 fixture 定位测试数据。
- Why：
  - 让集成测试可在任意 checkout 路径下运行。
- How：
  - 通过 `Path(__file__)` 相对仓库定位数据目录。
  - 若数据不入库，则把测试改为显式依赖环境变量，并在 skip reason 中给出清晰说明。
  - 将 `UNIT_TESTS.md` 中关于“CI 阻断/警告”的措辞与真实 CI 状态同步。

#### D2. 消除静默吞错

- 文件：
  - `src/dpeva/workflows/collect.py`
  - `src/dpeva/workflows/analysis.py`
  - 相应单元测试
- What：
  - 将 `except Exception: pass` 改为显式日志或窄化异常。
- Why：
  - 提升 Slurm 自提交和配置回填问题的可诊断性。
- How：
  - 优先识别 `config_path` 注入阶段的真实失败模式。
  - 若赋值失败只是类型/模型冻结问题，应记录 warning 并附带异常类型。
  - 为对应行为补回归测试，确保错误不会被静默吞掉。

## Implementation Steps

1. 先处理 Phase A，建立统一 Python 质量门禁并收敛 docs workflow。
2. 再处理 Phase B，使文档治理脚本与对外入口先恢复一致。
3. 然后执行 Phase C，统一 release helper、技能说明与脚本归宿。
4. 最后执行 Phase D，收敛测试便携性与源码异常处理。
5. 每完成一个 Phase，立即补对应验证，避免把多个治理层变更混在一起排错。

## Verification Steps

### 针对 CI / workflow

- 检查 workflow YAML 语义和触发路径是否覆盖：
  - `.github/workflows/python-quality.yml`
  - `.github/workflows/docs-check.yml`
  - `.github/workflows/doc-lint.yml`
  - `.github/workflows/docs.yml`
  - `.github/workflows/docs-deploy.yml`
- 本地执行与 workflow 对齐的命令：
  - `ruff check .` 或收敛后的明确路径
  - `pytest tests/unit`
  - `python3 scripts/audit.py src/dpeva`

### 针对文档治理

- `python3 scripts/doc_check.py`
- `python3 scripts/check_docs_freshness.py --days 90`
- `make -C docs html SPHINXOPTS="-W --keep-going"`

### 针对测试与源码

- `pytest tests/integration/test_labeling_rotation_bug.py -q`
- 与 `collect.py` / `analysis.py` 相关的单元测试或新增回归测试

## Risks

- `ruff check .` 目前可能被 `.trae/skills` 下现有问题阻断，因此必须先决定 lint 范围策略。
- 将 `plans`、`reports` 纳入文档治理后，短期内可能暴露大量 owner / freshness 存量问题，需要同步安排清理。
- 调整 docs workflow 触发条件后，CI 次数会短期上升，需要用缓存和路径过滤控制成本。
- 修复测试路径后，可能发现原测试数据本就未稳定入库，需要同步澄清数据来源策略。

## Acceptance Criteria

- 新的整改实现后，审查报告中的 F1-F8 全部有明确落点：已修复、已降级、或已转为可跟踪 backlog。
- 至少一条 Python 主线质量 workflow 在 PR 上生效。
- 活跃 `docs/plans/`、`docs/reports/` 进入治理脚本检查范围。
- 根 README 的文档入口不再包含占位链接与 `.rst` 源路径。
- release helper 脚本、skill 与 developer guide 的版本维护说明完全一致。
- 测试路径依赖与静默吞错问题完成收口并有验证证据。
