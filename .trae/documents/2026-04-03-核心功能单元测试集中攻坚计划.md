# 核心功能单元测试集中攻坚计划

## Summary

本计划面向 DP-EVA 当前剩余的核心功能单元测试薄弱区，聚焦 `analysis`、`inference`、`feature` 与 `labeling` 四条主工作流中的关键 manager / workflow 路径，目标是在不引入重型集成依赖的前提下，通过补齐负向、边界、降级与容错分支测试，显著提升核心功能的回归可信度与覆盖率质量。

本轮计划不再重复已经完成的基础治理工作（如 `analysis workflow` 收集修复、`postprocess` 第一轮补测、全局覆盖率门禁接入），而是在这些已完成基础上执行“第二阶段集中攻坚”。

## Current State Analysis

### 1. 已有基础

- 当前已接入 Python 质量工作流：`.github/workflows/python-quality.yml`
- 当前单元测试已具备整体覆盖率门禁：`pytest tests/unit --cov=src/dpeva --cov-branch --cov-fail-under=80`
- 已完成的关键治理包括：
  - `tests/unit/workflows/test_analysis_workflow.py` 收集结构修复
  - `tests/unit/conftest.py` 随机 fixture 固定种子
  - `tests/unit/feature/test_generator.py` 第一轮负向/边界补测
  - `tests/unit/labeling/test_postprocess.py` 第一轮分类/过滤/导出异常补测
  - `tests/unit/analysis/test_unified_analysis_manager.py` 第一轮慢图与绘图降级补测

### 2. 当前痛点

基于现有审查报告、最近几轮补测结果和仓库实际测试布局，当前剩余的主要薄弱区如下：

- `src/dpeva/analysis/managers.py`
  - `safe_plot()` 的慢图 warning 与绘图失败降级已开始覆盖，但分布图家族、cohesive energy、force magnitude、virial 分支仍有空洞。
- `src/dpeva/inference/managers.py`
  - `InferenceIOManager` 与 `InferenceExecutionManager` 中仍存在模型发现、composition 加载失败回退、模型缺失跳过、默认环境变量、脚本生成细节等低覆盖路径。
- `src/dpeva/workflows/feature.py`
  - 现有测试以 env / submission 主线为主，缺少 `data_path` 缺失、未知 mode、python/slurm 互斥、局部系统失败继续、递归输出契约等核心边界。
- `src/dpeva/workflows/labeling.py`
  - 当前主要覆盖 prepare/execute/extract/postprocess 的主流程与 stage log，缺少 `_load_dataset_map()` 多模式边界、execute retry、submit 失败继续、slurm 轮询异常、空任务/无 attempt 参数 等关键容错分支。

### 3. 当前测试文件分布

- `tests/unit/analysis/test_unified_analysis_manager.py`
- `tests/unit/inference/test_inference_execution_manager.py`
- `tests/unit/workflows/test_feature_workflow_env.py`
- `tests/unit/workflows/test_feature_workflow_submission.py`
- `tests/unit/workflows/test_labeling_workflow.py`
- `tests/unit/feature/test_generator.py`
- `tests/unit/labeling/test_postprocess.py`

### 4. 当前设计约束

- 必须坚持单元测试范式：优先 patch / mock 外部依赖，不执行真实 `dp`、`dpdata`、Slurm、ABACUS 或大规模文件系统递归。
- 新增测试应尽量围绕“契约断言”而不是实现细节，避免再次引入大量脆弱日志全文断言与 UI 魔法值断言。
- 覆盖率门禁已经在 CI 中设置为 `80%`；本轮不提高门禁阈值，而是先提升实测覆盖率与模块质量，避免流水线不稳定。

## Proposed Changes

### A. `tests/unit/analysis/test_unified_analysis_manager.py`

**What**
- 继续补 `UnifiedAnalysisManager.analyze_model()` 的剩余关键分支测试。

**Why**
- `src/dpeva/analysis/managers.py` 仍是核心低覆盖热点，且该模块承载统计、绘图、降级与性能提醒逻辑，是 analysis 主链路的核心风险点。

**How**
- 新增针对以下路径的定向测试：
  - `has_ground_truth=False` 时仅生成 predicted 分布图，不触发 metrics / parity 分支
  - `compute_relative_energy()` 返回 `None` 时 cohesive energy 全家桶被正确跳过
  - `force` / `virial` 为 `None` 与非 `None` 的组合分支
  - `plot_level="basic"` 与 `plot_level="full"` 对 overlay / with_error / enhanced parity 调用范围的差异
  - frame count mismatch 导致 composition info 自动禁用的 warning 分支
- 测试风格：
  - 通过 patch `StatsCalculator` 与 `InferenceVisualizer` 控制输入输出
  - 断言关键调用与 warning 语义，不做像素级或布局级断言

### B. `tests/unit/inference/test_inference_execution_manager.py`

**What**
- 补齐 `InferenceExecutionManager` 与必要的 `InferenceIOManager` 路径测试。

**Why**
- 当前该文件已覆盖基本提交循环，但仍缺少对命令构造、模型缺失跳过、local 默认 env、summary / statistics 保存等关键行为的系统守护。

**How**
- 在现有 `tests/unit/inference/test_inference_execution_manager.py` 基础上补充：
  - local backend 在 `env_setup=""` 时自动使用 `export OMP_NUM_THREADS=<n>`
  - models 列表中包含缺失文件时跳过并记录 warning
  - `task_name=""` 时输出目录回退到 `work_dir/<i>`
  - 生成脚本文件名在 `local/slurm` 两种 backend 下分别为 `run_test.sh` / `run_test.slurm`
  - command 中包含 `results_prefix`、`head` 与 `WORKFLOW_FINISHED_TAG`
- 视实际需要新增文件：
  - `tests/unit/inference/test_inference_io_manager.py`
- 若新增该文件，则补：
  - `discover_models()` 连续编号发现与遇空即停行为
  - `load_composition_info()` 在 `data_path` 缺失、`load_systems()` 抛错时返回 `(None, None)`
  - `save_statistics()` 对 numpy 类型序列化
  - `save_summary()` 空列表不写文件、非空写 CSV

### C. `tests/unit/workflows/test_feature_workflow_env.py` 与 `tests/unit/workflows/test_feature_workflow_submission.py`

**What**
- 补齐 `FeatureWorkflow` 的失败路径、模式边界与局部失败继续行为。

**Why**
- `src/dpeva/workflows/feature.py` 是 feature 主工作流入口，当前测试偏“happy path + submission”，尚未真正覆盖工作流编排中的错误与降级语义。

**How**
- 新增测试覆盖：
  - `data_path` 不存在抛 `WorkflowError`
  - `mode` 非 `cli/python` 时抛 `WorkflowError`
  - `backend=slurm` 且 `mode=python` 时走 worker 提交，不走本地递归
  - 多池 / 单池检测后传递给执行层的 `sub_pools` 契约
  - 递归过程中单个局部系统失败只记录 error，不中断整个批次
  - 默认 `savedir` 推导或显式 `savedir` 优先级
- 保持 mock 边界在 `FeatureExecutionManager`、`FeatureIOManager`、`DescriptorGenerator`

### D. `tests/unit/workflows/test_labeling_workflow.py`

**What**
- 对 `LabelingWorkflow` 做第二轮容错与边界补测。

**Why**
- 该工作流负责 prepare / execute / extract / postprocess 串联及 retry / slurm 轮询等关键容错逻辑，是 labeling 主链路的系统性风险点。

**How**
- 新增测试覆盖：
  - `_load_dataset_map()` 的三种检测模式：
    - root 直接 system
    - single-pool
    - multi-pool
  - `input_data_path` 不存在抛 `FileNotFoundError`
  - `attempt_params=[]` 时 fallback 到单次执行 warning 分支
  - `_submit_job_dirs()` 中部分 job 提交失败但其它 job 继续
  - `_monitor_slurm_jobs()` 在 `squeue` 查询失败时记录 error 并继续等待
  - `_collect_active_job_dirs()` 仅返回仍含子任务目录的 bundle
  - `run_execute()` 在没有 active jobs 时直接判定 all converged
- patch 重点：
  - `load_systems`
  - `dpdata.MultiSystems`
  - `subprocess.run`
  - `time.sleep`
  - `LabelingManager`

### E. `tests/unit/labeling/test_postprocess.py`

**What**
- 在现有第一轮补测基础上，继续补齐 `AbacusPostProcessor` 的核心计算与异常分支。

**Why**
- `src/dpeva/labeling/postprocess.py` 已被拉升一轮，但它仍承担数据载入、清洗、metrics 统计、导出与无贡献提示等核心行为，继续补测投入产出比高。

**How**
- 新增测试覆盖：
  - `_is_labeled_system_complete()` 各类短数组 / 缺字段返回
  - `compute_metrics()` 多 system、多 frame 混合输入
  - `ref_energies` 不完整时最小二乘补全 / 不补全分支
  - `pressure_gpa` 计算与空 cell / 非法体积时的行为
  - `process_directory()` 在无任务、全 failed、全 bad_converged、cleaned 空集等场景的 summary 输出

### F. 文档与门禁同步

**What**
- 对测试治理文档与报告做最小必要同步，保证计划实现后文档不漂移。

**Why**
- 当前 `tests/unit/UNIT_TESTS.md` 已同步到 `80%` 总体门禁；实施后若覆盖率或重点模块状态明显变化，应最小化更新说明，避免再次出现“代码已变、文档未变”。

**How**
- 仅在实现结果显著改变测试策略时更新：
  - `tests/unit/UNIT_TESTS.md`
  - `docs/reports/2026-04-02-Code-Review-Unit-Test-Audit.md`
- 若只是覆盖率数值提升而策略不变，可不改报告正文，仅在最终交付说明中汇总。

## Assumptions & Decisions

### Assumptions

- 继续以当前已接入的 `80%` 总覆盖率门禁作为实现约束，不在本轮把 CI 阈值抬高到仓库尚未稳定达到的数值。
- 现有测试命名与目录结构维持不变，以增量补测为主，不在本轮执行大规模重命名迁移。
- `analysis/managers.py` 的绘图测试仍采用 mock 驱动，不引入真实图片渲染基线。

### Decisions

- **决策 1：目标优先级**
  - 优先提升“核心功能回归可信度”而不是单纯追求全仓数值覆盖率。
  - 实施优先顺序固定为：
    1. `labeling/postprocess`
    2. `analysis/managers`
    3. `inference/managers`
    4. `workflows/feature`
    5. `workflows/labeling`

- **决策 2：覆盖率策略**
  - 本轮不调整 CI 门禁阈值。
  - 成功标准是“在维持门禁通过的前提下，使上述核心热点模块的覆盖率显著提升，并减少未覆盖的容错/降级分支”。

- **决策 3：测试风格**
  - 所有新增测试都遵循：
    - Mock 外部依赖
    - 断言行为契约
    - 避免全文日志和脆弱 UI 数值
    - 使用 `tmp_path`，避免硬编码绝对路径
