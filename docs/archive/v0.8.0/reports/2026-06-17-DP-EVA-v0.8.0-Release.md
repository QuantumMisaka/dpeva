# DP-EVA v0.8.0 Release Notes

- Status: active
- Audience: Users / Developers / Maintainers
- Last-Updated: 2026-06-17
- Version: 0.8.0
- Branch: `develop`
- Release candidate base: `0e86d34`
- Final release point: current `develop` after this release-readiness changeset is pushed
- DOI: <https://doi.org/10.1093/ce/zkag029>

DP-EVA v0.8.0 is the first release candidate after the project article went online in *Clean Energy*. This release focuses on three themes: removing fragile external runtime bindings, introducing optional exploration support through `atst-tools`, and landing the first DP-EVA-native energy-level LLPR/DPOSE uncertainty path for DPA/DeepMD workflows.

The release is validated on the `dpeva-dpa4-test` environment. At the time of this report, the source version is `0.8.0`, `develop` has been validated with the current release-note and exploration recipe updates, and the `v0.8.0` Git tag has not yet been created.

## Highlights

### Project Citation and Publication Metadata

The DP-EVA article is now online:

> Zhaoqing Liu, Zhe Deng, Huabo Zhao, Han Wang, Mohan Chen, Hong Jiang, DP-EVA: Data-Efficient Fine-Tuning Framework via Maximizing Pre-Trained Knowledge of Large Atomistic Models for Developing Domain-Specific Machine Learning Interatomic Potentials, *Clean Energy*, 2026, zkag029, <https://doi.org/10.1093/ce/zkag029>.

Repository metadata now includes:

- `CITATION.cff` with a preferred article citation.
- A README citation section with BibTeX.
- A `Paper` URL in `pyproject.toml`.
- Documentation entry points that route users to the DOI and citation instructions.

### Labeling: Internal ABACUS Writer

v0.8.0 removes the runtime dependency on the GitLab `ase-abacus` fork for the standard labeling prepare path. DP-EVA now owns a minimal internal ABACUS writer for `INPUT`, `KPT`, and `STRU`, with golden baseline tests to protect output format stability.

This reduces environment fragility for SAI/HPC workflows and makes the labeling workflow easier to test in a standard ASE environment.

### Exploration: Optional ATST Backend

This release adds the `dpeva.exploration` abstraction layer, an `ATSTToolsBackend`, and the `dpeva explore` CLI entry point. The first supported backend paths are `md` and `relax`.

`atst-tools` remains optional and is installed through:

```bash
pip install -e '.[explore]'
```

The backend records DP-EVA-side input structure snapshots and writes a `dpeva_exploration_result.json` manifest for successful and failed runs.

The default example recipe is now MD-based, matching the common active-learning pattern where exploration generates candidate configurations through molecular dynamics driven by an ML potential:

```bash
cd examples/recipes/exploration
dpeva explore config_explore_md.json
```

The DP-EVA CLI still consumes a JSON wrapper. The backend-native ATST YAML is referenced from that wrapper through `backend_config_path`.

### Energy-Level LLPR/DPOSE Uncertainty

v0.8.0 introduces a DP-EVA-native analytic energy LLPR path based on DeepMD fitting last-layer features. The implementation provides covariance construction, calibration helpers, collect-ready energy uncertainty columns, and DPOSE ensemble data structures when real last-layer weights are supplied.

The release deliberately keeps force-level DPOSE guarded. Detached last-layer feature arrays cannot support differentiable force propagation, so `force` and `energy_force` LLPR/DPOSE targets fail fast until a validated DeepMD internal PyTorch adapter exists.

### Documentation, Governance, and CI

The release tightens documentation lifecycle rules and keeps active reports separate from archived process records. Coverage reports are now generated under `build/coverage/coverage-unit.json`, so local test runs no longer modify root-level tracked coverage artifacts.

CI and local validation now cover:

- Ruff checks for `src`, `tests`, and `scripts`.
- Unit tests with branch coverage and an 80% coverage gate.
- Documentation governance and freshness checks.
- Sphinx documentation build with warnings treated as errors.
- Optional exploration smoke checks for `dpeva explore --help` and `atst --help`.
- README and recipe examples for `dpeva explore` now use a local recipe-directory command and the MD exploration wrapper instead of a root-relative relax wrapper path.

## Upgrade Notes

- Core installation remains:

  ```bash
  pip install -e .
  ```

- Exploration support is optional:

  ```bash
  pip install -e '.[explore]'
  ```

- `ase>=3.28.0` is now the project baseline.
- `atst-tools>=2.1.0` is not a core dependency; install it only for exploration workflows.
- The documented exploration example is `examples/recipes/exploration/config_explore_md.json`, which points to backend-native `atst_md.yaml`.
- The release keeps the main workflow boundary unchanged: `train`, `infer`, `feature`, `collect`, `label`, `analysis`, and `clean` remain independent CLI workflows.
- Automatic production wiring from exploration into `feature -> UQ -> DIRECT -> label` is intentionally left for a later 0.8.x design.

## Known Boundaries

- `v0.8.0` tag creation is still pending in this working copy.
- Energy LLPR is usable as an uncertainty ranking/calibration backend, but complete DPOSE shallow-ensemble parity requires automatic extraction of real DeepMD last-layer weights.
- Force-level DPOSE is not exposed to users in this release.
- Exploration is an independent workflow in v0.8.0; formal active-learning loop wiring is a future 0.8.x task.
- The MD exploration recipe is a minimal wrapper example. Production ATST MD settings remain the responsibility of the backend-native YAML and user runtime environment.
- Historical v0.8.0 planning and audit materials remain in `docs/archive/v0.8.0/`.
